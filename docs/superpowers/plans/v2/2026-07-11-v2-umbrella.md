## 各 sub-plan 简述

### #1 冲突对比页 ConflictView

**问题：** 当前 InboxView 的「跳过」按钮把冲突留在 inbox 永远走不掉；「V2: 内容比对」是死按钮。

**后端新增：**
- `GET /api/conflicts/:id/compare` → 返回 `{ a: { files: [..], cover_url }, b: { files: [..], cover_url } }`，从 zip 抽文件清单（不解压到磁盘）
- 已有 HTTP 服务层加 handler，复用 `archive::list_images`

**前端新增：**
- `ConflictView.vue`：`/inbox/compare/:id`，左右两栏对比（封面 + 文件列表）
- 三个动作：保留 A / 替换为 B / 都保留 / 都跳过
- 路由表 + store 新增 actions

**验收：** 拖两个同名不同 hash 的 zip 进 inbox，对比页能列出两个 zip 内部图片清单，按钮能把状态改成 identified 或 leave-in-inbox。

### #2 设置页增强 + HTTP token 鉴权

**后端：**
- `port_allocator` 模块：保留端口失败时按 100ms × N 退避重试，超过 N 次才回退到随机
- `auth` 中间件：读 `app_setting.auth_token`（首次启动生成随机 token 落库），校验 `Authorization: Bearer <token>` 头
- `/api/health` 豁免 auth（健康检查不该卡 token）

**前端：**
- SettingsView 增 3 个区块：
  - **HTTP 端口**：固定端口输入框 + 「随机」开关（默认随机）
  - **Token**：显示当前 token + 「重新生成」按钮 + 复制按钮（首次启动生成）
  - **Inbox 目录**：可改路径 + 「重新加载」按钮
- 所有 HTTP 调用统一通过 `apiClient` 包装，自动注入 token

**验收：** 浏览器扩展首次调用需要 token 才能拿到结果；端口被占时自动重试 3 次；token 重新生成后老 token 立刻失效。

### #3 三个 V2 HTTP 路由

| 路由 | 用途 | 实现路径 |
|---|---|---|
| `GET /api/doujinshi/check?hash=<blake3>` | 浏览器扩展"这本我下载过没？" | 复用现有 `by_hash` handler，加 query alias |
| `POST /api/doujinshi/:id/viewed` | 标记看过 | 新增 commands::library 的 `mark_viewed`，HTTP 桥接 |
| `GET /api/covers/by-hash/<hash>` | 按 hash 直接拿封面 | `by_hash` + `cover` handler 串联 |

**验收：** 三个端点都通过 `http_routes` 集成测试覆盖，包含 auth token header。

### #4 CI 加 `tauri build` job（deferred）

`.github/workflows/ci.yml` 新增第三个 job：

```yaml
tauri-build:
  runs-on: windows-latest
  steps:
    - uses: actions/checkout@v4
    - uses: dtolnay/rust-toolchain@stable
    - uses: Swatinem/rust-cache@v2
      with: { workspaces: src-tauri -> target }
    - uses: pnpm/action-setup@v4
      with: { version: 9 }
    - uses: actions/setup-node@v4
      with: { node-version: 20 }
    - uses: actions/cache@v4
      with:
        path: ~/AppData/Local/pnpm/store
        key: pnpm-${{ runner.os }}-${{ hashFiles('pnpm-lock.yaml') }}
    - run: cd src-tauri && cargo test  # 复用
    - run: pnpm install --frozen-lockfile
    - run: pnpm tauri build --bundles nsis  # 先 NSIS 一个，时间好估
    - uses: actions/upload-artifact@v4
      with:
        name: tauri-nsis
        path: src-tauri/target/release/bundle/nsis/*.exe
```

**验收：** workflow 在 windows-latest 上稳定出 NSIS 包，artifact 可下载。

### #7 RAR 完整解压

**方案：** A（调系统 unrar.exe），带前置检查 + 大小分级 + 空间保护（用户中午确认）。

#### 前置检查（必须先过，否则阻止入库）

1. **探测 unrar.exe 路径**，按顺序找：
   - 环境变量 PATH
   - `C:\Program Files\WinRAR\unrar.exe`
   - `C:\Program Files (x86)\WinRAR\unrar.exe`
   - `C:\Program Files\7-Zip\7z.exe`（备选，7z.exe 也支持 RAR 解压）
2. **找不到**：停止后续入库逻辑（不 hash、不移文件、不入 DB），InboxView 显示错误卡片：
   - 文案：「本机未安装 RAR 解压工具，请先安装 WinRAR 或 7-Zip」
   - 提供两个按钮：「下载 WinRAR」「下载 7-Zip」（链官网）

#### 大小分级（阈值用户拍板 200 MB / 1 GB）

- **小文件（< 200 MB）**：直接解压，后台异步
- **中等（200 MB ~ 1 GB）**：弹窗「文件较大 (XX MB)，确认解压？」用户点确认才继续
- **超大（> 1 GB）**：直接拒绝，错误卡片显示「文件过大 (XX MB)，超过最大解压限制 1 GB」

#### 空间保护

- 用 `unrar l` 或 `unrar v` 拿解压后未压缩大小
- 用 `sysinfo::Disks` 查当前盘剩余空间
- 解压后大小 > 剩余空间 → 拒绝解压，错误卡片显示「解压后预估需 XX MB，磁盘剩余 YY MB」

#### 实现路径

- `services/archive.rs` 新增 `extract_rar(path, dest)` → 返回 `Result<ExtractStats>`，调用 unrar 子进程
- 新增 `services/rar_detect.rs` → 探测 unrar.exe 路径，返回 `Option<PathBuf>`
- `services/disk_space.rs` → 查剩余空间
- `identifier.rs` 在 RAR 分支调上述三件套
- 前端 `InboxView` 增错误状态展示（标识符失败原因）

#### 验收

- 装 WinRAR → 拖小 RAR → 完整入库（hash + 文件清单 + 抽封面）
- 拖中等 RAR → 弹窗确认 → 确认后入库
- 拖超大 RAR → 卡片显示「文件过大」
- 卸载 WinRAR → 拖 RAR → 卡片显示「请先安装 WinRAR 或 7-Zip」+ 下载按钮
- 磁盘剩余 < RAR 内部大小 → 卡片显示「磁盘空间不足」
