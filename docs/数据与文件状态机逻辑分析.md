**现状梳理（数据 ↔ 文件 ↔ 状态机）**
- **数据是真实来源**：主表 `doujinshi` 里用 `current_location` 表达状态（至少有 `identified / will_delete / archived / permanently_deleted`），同时还有 `current_path` 与 `has_physical_file`。实现见 [doujinshi_file.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/db/entities/doujinshi_file.rs)。
- **状态机是“强一致迁移”**：`identified↔archived/will_delete` 这些迁移要求源文件必须存在、目标目录不能冲突；否则直接报错且 **DB 不改**。`permanently_deleted` 会 best-effort 删除文件（不存在也算成功），然后把 DB 写成 `permanently_deleted` 且 `has_physical_file=false`。见 [state_machine.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/services/state_machine.rs#L1-L161)。
- **自动化只有 inbox→入库**：inbox 扫描器会自动识别压缩包并入库到 `identified`，含冲突处理/去重/抽封面等。见 [scanner.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/services/scanner.rs#L37-L208)、[identifier.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/services/identifier.rs#L91-L426)。
- **扫描缺失文件不会改“状态机状态”**：启动 `dirty_scanner` 发现 DB 指向缺失文件时，会把 `has_physical_file=false` 并写一条 `dirty_data`，但**不会改 `current_location`**。这点与你说的“文件没了只反馈，不影响数据状态机”是基本一致的。见 [dirty_scanner.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/services/dirty_scanner.rs#L1-L203)。
- **`permanently_deleted` 的现实用途**：现在主要用于两类场景  
  1) 回收站“永久删除”命令会把 will_delete → permanently_deleted；见 [recycle.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/commands/recycle.rs#L77-L94)  
  2) inbox 冲突 “ReplaceB” 会把 A 推进 permanently_deleted 来释放同名文件占用；见 [inbox.rs](file:///d:/NewCode/doujinshi-records/src-tauri/src/commands/inbox.rs#L96-L176)

**与你的想法对齐/不对齐点**
- 对齐：扫描发现文件缺失目前确实只打标（`has_physical_file=false` + `dirty_data`），不改变 `current_location`。
- 不对齐：状态机里不少迁移是“文件必须存在，否则不让改 DB”。而你倾向于“人想把数据改成什么状态就是什么状态；文件存在则尽力搬/删，不存在就不动文件”——这会要求状态机从“强一致”改成“DB 状态为主、文件操作 best-effort + 约束校验/反馈”的模型。
- 不对齐：`permanently_deleted` 现在被当作一种“最终态”，并参与冲突处理逻辑；你认为它多余，更像“文件附庸”属性。

**现状 vs 你的目标**

**可选方案（2-3 个）**
- 方案 1（推荐，改动最小）：**保留“状态字段 current_location”，但把状态机改成“DB 优先 + 文件 best-effort”**
  - `permanently_deleted` 直接改语义/改名为 **`deleted`（已删除）**：它只是一个逻辑状态，不再代表“盘上一定不存在”（盘上不存在用 `has_physical_file` 表示）。
  - 状态迁移：如果文件存在，就执行移动/删除并校验冲突；如果文件不存在，就跳过文件操作但仍允许状态变更，同时把 `has_physical_file=false`、记录一条事件/脏数据用于提示。
  - 优点：最贴近你“状态是人的决定”；实现成本低；现有表结构基本不动。
  - 缺点：`current_location` 既表达“业务状态”又隐含“期望目录”，仍然有点耦合，但可接受。
- 方案 2（更干净，改动更大）：**把“业务状态”和“文件意图/期望”拆成两个字段**
  - `status = inboxed/archived/deleted/...`（纯业务）
  - `file_expectation = should_exist | may_be_missing | confirmed_absent`（纯文件语义）
  - 优点：模型最清晰，避免状态=目录的历史包袱。
  - 缺点：迁移/前后端改动更大，你现在的诉求可能不需要这么重。
- 方案 3（不推荐）：事件溯源/多表状态历史，太重，不符合当前项目规模。

我建议用 **方案 1**：把 `permanently_deleted` 收敛成你要的“已删除”，并把所有迁移从“强一致”改成“文件存在才操作文件、否则只更新 DB + 反馈”。

**方案 1 的设计（请你确认）**
- **状态集合（current_location）**
  - `identified`：入库（可预览/可正常操作；但允许 `has_physical_file=false`）
  - `archived`：归档（文件通常在 archived 目录，但允许用户拿走导致缺失）
  - `will_delete`：回收站（集中放文件，待用户决定是否销毁）
  - `deleted`：已删除（逻辑上已删除；允许恢复为 identified，但文件不会“恢复”，只是数据回到入库态）
- **文件存在性（has_physical_file）**
  - 纯观测值：扫描/操作时更新。
  - 任何状态都允许 `has_physical_file=false`（这正是你要的“文件没了也不影响状态”）。
- **迁移规则（关键点）**
  - MarkForDelete（任意 → will_delete）：  
    - 有文件：移动到 will_delete 目录（需要目标冲突检查）  
    - 无文件：仅更新状态为 will_delete，并产生“文件缺失”的反馈记录
  - Destroy（will_delete/identified → deleted）：  
    - 有文件：删除（best-effort）  
    - 无文件：直接标记 deleted（同样记反馈）
  - Restore（deleted/will_delete/archived → identified）：  
    - 有文件：尽力移动回 identified（仍做冲突检查）  
    - 无文件：只把状态改回 identified（便于你“重新喜欢/重新整理”，但预览不可用）
- **扫描行为**
  - dirty_scanner：继续“只标记缺失/不改状态”，并把缺失作为可展示的反馈来源（现在已有 dirty_data/scan_event 基础）。
